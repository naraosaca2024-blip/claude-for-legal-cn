# /is-this-a-problem

1. 加载 `~/.claude/plugins/config/claude-for-legal/product-legal/CLAUDE.md` → 风险校准。
2. 运行以下分流工作流。
3. 模式匹配。检查常见陷阱。
4. 一分钟内回答：✅ 可以 / ⚠️ 需要看一下 / 🛑 暂停。一句话原因。
5. 如果 ⚠️ 或 🛑：命名下一步。

```
/product-legal:is-this-a-problem "我们可以在定价页上使用客户 logo 吗？"
```

## 事项上下文

**事项上下文。** 检查执业级 CLAUDE.md 中的 `## Matter workspaces`。如果 `Enabled` 为 `✗`（内部用户的默认值），跳过本段的其余部分——skills 使用执业级上下文，事项机制不可见。如果启用且没有活跃事项，询问："这是哪个事项的？运行 `/product-legal:matter-workspace switch <slug>` 或说 `practice-level`。"加载活跃事项的 `matter.md` 以获取事项特定上下文和覆盖。将输出写入事项文件夹 `~/.claude/plugins/config/claude-for-legal/product-legal/matters/<matter-slug>/`。除非 `Cross-matter context` 为 `on`，否则永远不要阅读另一个事项的文件。

## 目的地检查

在生成输出之前，检查它要去哪里。如果用户命名了目的地（频道、分发列表、对手方、"每个人"），询问它是否在特权圈内。公共频道、公司范围列表、对手方/对方律师、供应商和客户（对于工作产品）放弃保护。当目的地看起来在圈外时，标记它并提供（a）仅限律师使用的特权版本，（b）更广泛频道的清理版本，或（c）两者——不要默默应用特权标题然后帮助将其粘贴到标题不保护它的地方。请参阅此 plugin 的 CLAUDE.md 中的规范 `## Shared guardrails → Destination check`。

## 目的

大多数"快速法律问题"Slack 是三种情况之一：（a）不是问题，快速告知，（b）真正需要真正审查的事情，路由它，（c）看起来可以但有陷阱的事情，捕获陷阱。此 skill 使用校准表在一分钟内完成分流。

目标是速度。PM 在下午 4:47 提问。他们要的是答案，不是备忘录。

## 加载校准

读取 `~/.claude/plugins/config/claude-for-legal/product-legal/CLAUDE.md` → `## Risk calibration`。此 skill 的全部意义就是针对该表进行模式匹配。

## 分流

### 对校准进行匹配

问题是否匹配校准表中的模式？

**匹配"通常 FYI"：**
→ 告知。一行。"没问题——[模式]。发布吧。"

**匹配"通常需要工作"：**
→ 命名工作。"需要 [PIA / 供应商审查 / 声明检查]。需要 [表中的时间线]。要我启动吗？"

**匹配"通常阻塞"：**
→ 阻止他们。"暂停——[模式]。在任何人承诺日期之前，这需要真正审查。我们谈谈。"

**不匹配任何内容：**
→ 也告知。"这没有模式匹配到我见过的任何内容。需要人工查看——[你的名字] 或我明天？"

### 陷阱检查

有些问题表面看起来没问题但有弯。识别事实模式，提出陷阱问题，然后在得出是否是问题的结论之前研究适用法理对于特定事实的适用。

| 问题听起来像 | 为何可能不简单 | 陷阱问题 |
|---|---|---|
| "我们可以将 [供应商] 添加到集成中吗？" | 供应商触及新数据类别——标记为可能涉及隐私和供应商风险制度并路由研究 | "什么数据流向他们？" |
| "我们可以对定价页做 A/B 测试吗？" | 按细分差异定价可能涉及消费者保护和反歧视制度——标记并路由研究 | "两个分支看到相同东西的价格相同吗？用户如何分配到各分支？" |
| "我们可以自动注册用户使用新功能吗？" | 对之前选择退出的用户默认开启行为可能涉及同意和消费者保护规则——标记并路由研究 | "这是否尊重现有偏好？" |
| "我们可以在网站上使用客户 logo 吗？" | Logo 使用是与合同关系分开的许可——标记为可能涉及宣传/背书规则和客户自己的合同条款 | "合同中关于宣传说了什么？我们有书面许可吗？" |
| "我们可以用这些数据训练吗？" | 原始收集目的的使用权可能不扩展到训练——标记并研究收集时给用户的通知/同意 | "收集时我们告诉用户什么？用户在哪些司法管辖区？" |
| "这只是内部工具" | 内部工具仍然处理个人数据——标记为可能涉及隐私制度并路由研究 | "它涉及谁的数据？员工、客户、第三方？" |
| "我们已经做了类似的事情" | "类似"做了很多工作——差异通常是问题所在 | "类似在哪里？实际有什么不同？" |
| "我们可以用 [AI 供应商 / LLM] 做这个吗？" | 供应商 AI 条款可能允许在输入上训练；用例可能需要 AIA——标记并路由到 `/ai-governance-legal:use-case-triage` | "有 AI 附录吗？什么数据进入模型？" |
| "我们可以给这个功能加 AI 吗？" | 可能是注册表中没有的新用例；可能触发 AIA 要求——标记并路由到 `/ai-governance-legal:use-case-triage` | "AI 做什么——辅助还是自动化？它作用于谁？" |
| "模型只是自动决定" | 没有人工审查的自动化决策在某些司法管辖区受监管——标记并研究受影响用户司法管辖区适用的规则 | "谁受影响？有人的参与吗？受影响的用户在哪里？" |
| "这是 AI 生成的内容" | 输出 IP 和披露义务因司法管辖区和供应商条款而异——标记并路由研究 | "什么内容类型？供应商的 ToS 是否涉及输出所有权？受众是谁？" |
| "我们只是在我们的数据上微调" | 训练数据权利、输出 IP 和供应商义务都变了——标记并路由到 `/ai-governance-legal:vendor-ai-review` | "训练数据中有什么？其中有没有客户或员工数据？" |

如果可能存在陷阱，在回答之前提出一个问题。一个问题，不是检查清单。当答案表明存在真正问题时，标记以供研究并路由——不要仅从问题本身模式匹配到法律结论。

## 输出格式

**对于 Slack（常见情况）：**

Slack 分流回复是内部法律建议。如果回复被粘贴到与非法律人员广泛共享的工单、文档或频道中，请在前面加上 `~/.claude/plugins/config/claude-for-legal/product-legal/CLAUDE.md` `## Outputs` 中的工作产品标题（因用户角色而异——见 `## Who's using this`）：

```
[WORK-PRODUCT HEADER — per plugin config ## Outputs]
```

对于流程中的 Slack DM 回复给 PM，简短形式为：

```
[✅ 可以 | ⚠️ 需要看一下 | 🛑 暂停]

[一句话：判断和原因。]

[如果 ⚠️：看一下涉及什么，需要多长时间]
[如果 🛑：找谁谈，何时]
```

**示例：**

```
✅ 可以——添加分析事件在这里是 FYI，只要它被现有隐私政策类别覆盖。这个是。
```

```
⚠️ 需要 PIA——[类别] 的新数据收集。通常需要一天。
要我启动吗？
```

```
🛑 暂停——"在客户数据上训练"触发一堆事情。客户协议中关于数据使用说了什么？在任何人向客户承诺之前，我们先拉出来看看。
```

```
⚠️ 需要 AI 治理分流——在此工作流中添加 LLM 意味着我们需要对照注册表检查用例并确认 AIA 在发布前完成。需要一天。要我运行 `/ai-governance-legal:use-case-triage` 吗？
```

## 何时不使用此 skill

- 问题实际很复杂（多个问题、新颖领域）→ 路由到 launch-review 或 feature-risk-assessment
- 问题是"你能审查这个 PRD 吗" → 那是 launch-review，不是分流
- 你不确定 → 说"我不确定，让我正式查看"——错误的快速答案比缓慢的正确答案更糟

## 语气

快速、直接、有用。PM 不是在请求演讲。如果没问题，说"可以"——不要列出你检查的七件事。如果有问题，说哪里有问题以及该怎么办。

你是人们想要询问的律师，而不是他们绕过的律师。
