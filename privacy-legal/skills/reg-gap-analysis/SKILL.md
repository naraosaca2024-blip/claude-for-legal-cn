# 法规到政策差距分析

## 目的

一个州通过了一部新的隐私法。ICO 发布了新指导。CPPA 最终确定了法规。某些东西在变化——现在你需要知道你必须改变什么，如果有的话。

此 skill 将新要求与你当前的做法进行差异比较（根据 `~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` → 隐私政策承诺 + PIA 中记录的实践），并生成带有整改计划的差距列表。

## 加载当前状态

读取 `~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md`：
- `## Privacy policy commitments` — 你公开承诺的内容
- `## Regulatory footprint` — 已经适用的内容
- `## DSAR process` → 系统列表 — 你实际在操作上做了什么

如果法规对你不适用（错误司法管辖区、低于阈值、不同行业），差距分析只有一行："不适用。原因：[原因]。无需行动。"

## 工作流

### 步骤 1：确定法规范围

在差异比较之前，回答：

- **是否适用？** 司法管辖区（你在那里有数据主体吗？）、阈值（收入、用户数量、数据量）、行业豁免
- **何时？** 生效日期、执法日期（通常更晚）、任何分阶段
- **什么是真正新的？** 许多"新"的州隐私法是 90% CCPA 加上调整。识别与你已经遵守的内容的差异，而非全文。

### 步骤 2：提取要求

阅读法规（或摘要/指导）。将每个实质性要求列为离散项目：

| # | 要求 | 引用 | 类别 |
|---|---|---|---|
| 1 | [要求原文] | [章节] | [通知 / 权利 / 安全 / 供应商 / 其他] |

**类别：**
- **通知** — 你必须告诉用户什么（隐私政策内容）
- **权利** — 用户可以请求什么（DSAR 相关）
- **安全** — 技术/组织措施
- **供应商** — 你必须向处理者传达什么
- **同意** — 选择加入/退出机制
- **治理** — DPO、影响评估、记录保存

### 步骤 3：与当前状态差异比较

对于每个要求：

```markdown
### [要求 #N]：[简称]

**法规要求：** [要求，引用或复述]

**我们目前：** [配置 CLAUDE.md / 隐私政策 / 实践显示的内容]

**差距：** [无 | 部分 | 完全]

**如果是部分/完全差距——缺少什么：** [具体]

**关闭所需努力：** [仅政策更新 | 产品变更 | 供应商重新谈判 | 新流程]

**不合规风险：** [监管处罚范围、执法可能性、声誉]
```

### 步骤 4：优先级排序

不是每个差距都一样。按以下排序：

1. **有实际约束力的硬截止日期** — 生效日期 + 活跃执法 + 真实处罚
2. **努力与影响比** — 政策语言更新成本低；产品重建不低
3. **你已经做了多少** — 如果你对 GDPR 已经 80%，州法增量可能很小

### 步骤 5：整改计划

在前面加上 `~/.claude/plugins/config/claude-for-legal/privacy-legal/CLAUDE.md` `## Outputs` 中的工作产品标题（因用户角色而异——见 `## Who's using this`）。

> **研究连接器飞行前检查。** 在发出整改计划之前，检查法律研究连接器在此会话中是否可达——Westlaw、EUR-Lex / 监管机构站点连接器或任何公司配置的研究 MCP。将此收集到审阅者备注中，根据 CLAUDE.md `## Outputs`：如果没有连接器在步骤 2 或常见法规类别研究步骤中返回结果（或在运行时未配置），在审阅者备注的 **Sources:** 行中记录——例如，`未连接——来自训练知识的引用；隐私差距分析中最高伪造风险的项目是新州法生效日期、执法开始日期和条款/章节精确定位——优先检查这些`。每个引用的 `[model knowledge — verify]` 标记保持内联。不要在输出上方发出独立横幅。

```markdown
[WORK-PRODUCT HEADER — per plugin config ## Outputs]

## 整改计划：[法规名称]

**生效日期：** [日期]
**执法开始：** [日期]

### 执法前必须完成

| 差距 | 修复 | 负责人 | 截止 | 状态 |
|---|---|---|---|---|
| [差距] | [具体修复] | [姓名] | [日期] | [ ] |

### 应该完成（风险较低，不阻塞）

[相同表格]

### 已合规

[差距 = None 的要求列表——有助于"我们基本没问题"的消息]

### 接受的差距（风险已接受，不修复）

[如有——附有记录的理由和谁接受了风险]
```

## 常见法规类别

在确定增量范围时，将新法规大致归入一个类别然后研究具体内容是有帮助的：

- **基线数据保护/隐私法** — 对某个司法管辖区个人数据实践的广泛覆盖
- **行业特定覆盖** — 健康、金融、儿童、教育、就业等
- **AI 特定制度** — 自动决策的透明度、影响评估或治理
- **数据经纪商/广告技术制度** — 注册、退出、删除机制
- **违规通知制度** — 独立或嵌入在更广泛的法律中
- **跨境传输制度** — 充分性、机制和评估要求

对于与新法规相关的每个类别，**在起草差距分析之前研究当前有效的要求**。引用主要来源。验证时效性——新州法在每个立法会议期间上线，监管机构发布改变给定控制"合规"含义的解释性指导。标记不确定性以供律师验证，而非断言你未确认的规则。

> **没有静默补充。** 如果对配置的法律研究工具的搜索查询对于法规、指导文件或执法行动返回很少或没有结果，报告发现的内容并停止。不要在未询问的情况下从网络搜索或模型知识填充空白。说："搜索从 [工具] 返回了 [N] 个结果。对于 [制度/主题] 的覆盖范围似乎很薄。选项：（1）扩大搜索查询，（2）尝试不同的研究工具，（3）搜索网络——结果将标记为 `[web search — verify]` 并应在依赖之前根据发布机构进行检查，或（4）标记为未验证并停止。你想要哪一个？"律师决定是否接受较低置信度的来源。
>
> **来源归因分层。** 用其来源标记差距分析中的每个引用。对于模型知识引用，使用三个层级之一，而非单一的全局"verify"标记：
>
> - `[settled]` — 稳定、众所周知的法定和监管参考，不太可能改变（例如，GDPR Art. 33、CCPA § 1798.100、FTC Act § 5）。在提交前仍需验证，但优先级较低。
> - `[verify]` — 真实但应验证的模型知识引用：具体实施法规、机构指导、案例裁决、阈值、生效日期、新颁布的州法规。
> - `[verify-pinpoint]` — 精确定位引用（具体小节字母、卷/页码、段落号、监管子部分引用）具有最高伪造风险，应始终根据主要来源验证。
>
> 工具检索的引用保留其来源标记（`[Westlaw]`、`[issuing authority site]` 或 MCP 工具名称）；网络搜索引用保持为 `[web search — verify]`；用户提供的引用保持为 `[user provided]`。分层显示真正的验证工作——验证一切的读者什么都没有验证。永远不要剥离或折叠标记。

## 与其他 skills 的集成

**来自 PIA 生成：** PIA 标记隐私政策不一致 → 这些作为已知差距输入此处。

**到 regulatory-legal 插件（如已安装）：** 此 skill 是手动版本。监控插件监视订阅源并在某些内容变化时自动触发此分析。

## 输出

保存为带日期的 markdown 文档。整改计划表成为跟踪器——随项目关闭更新状态。

如果差距分析结论为"无差距，我们合规"，仍要写文档——这在你后来被问到时是有用证据，表明你查过了。

**以引用验证备注结束：**

> 此输出中的引用由 AI 模型生成，尚未根据主要来源验证。在依赖任何法规、法律、指导或执法行动之前，根据法律研究工具（Westlaw、你事务所的研究平台或发布机构的网站）检查其准确性和当前状态。AI 生成的引用有时会被伪造或误引。每个引用上的来源标记（例如，`[web search — verify]`）显示其来源；`verify` 标记携带更高伪造风险，应首先检查。

## 以下一步决策树结束

根据 CLAUDE.md `## Outputs` 以下一步决策树结束。根据此 skill 刚刚生成的内容自定义选项——五个默认分支（起草 X、升级、获取更多事实、观察等待、其他）是起点，而非锁定。树就是输出；律师选择。

## 此 skill 不做什么

- 它不权威地解释模糊的监管语言。当法规不清晰时，说明："第 X 条可以被解读为 [A] 或 [B]。[A] 是保守解读。如果这很重要，建议外部律师。"
- 它不主动跟踪监管变化。它在你指向变化时运行。对于主动监控，参见 regulatory-legal 插件。
- 它不实施修复。它规划修复。
